2. Cloudflare Stream — How It Works
ℹ️
Cloudflare Stream is a fully managed video hosting, transcoding, and streaming service. It is used instead of S3/R2 for video files because it automatically transcodes videos to HLS (adaptive bitrate), generates thumbnails, delivers via global CDN, and supports signed playback URLs to prevent hotlinking. The backend never stores video files — it only stores Cloudflare Stream metadata (video ID, playback URL, duration, etc.).
Complete Video Upload Flow (Step by Step):
User picks or records a video in the Expo app using expo-image-picker or expo-camera.
The Expo app calls the backend: POST /api/v1/videos/upload/request. The backend calls the Cloudflare Stream API to create a direct upload URL. Cloudflare returns a one-time upload URL and a Cloudflare video ID.
The backend saves a video record in the database with status: 'pending' and returns the upload URL and video ID to the Expo app.
The Expo app uploads the raw video file directly to the Cloudflare upload URL using the TUS resumable upload protocol (tus-js-client or fetch with Content-Type: video/*). The video goes straight from the user's device to Cloudflare — it never passes through the Django server. This keeps the Django server fast and avoids bandwidth costs.
Cloudflare receives the video and begins transcoding it into multiple HLS resolutions (360p, 480p, 720p, 1080p). This takes between 30 seconds and a few minutes depending on video length.
Cloudflare sends a webhook event to POST /api/v1/webhooks/cloudflare-stream when transcoding is complete (status: ready) or fails (status: error).
The backend webhook handler updates the video record in the database: sets status to 'ready', saves the playback URL, thumbnail URL, and duration.
The Expo app polls GET /api/v1/videos/{video_id}/status every 3 seconds until status changes from 'processing' to 'ready', then shows the video as live.
Viewers stream the video via the HLS playback URL from Cloudflare's CDN. The backend issues signed playback tokens for premium/locked videos to prevent unauthorised access.
Video Visibility States:
public — Visible to all users in the For You feed and on the creator's profile.
followers_only — Visible only to users who follow the creator.
premium — Locked. Viewers must have an active monthly subscription to the creator to watch (covered in Tipping & Wallet doc).
private — Visible only to the creator. Does not appear in any feed.
processing — Temporary state while Cloudflare transcodes the upload. Not visible to viewers.
failed — Transcoding failed. Visible only to the creator with a retry option.
removed — Taken down by admin moderation. Not visible to any user.
Cloudflare Stream Signed Playback Tokens (for Premium Videos):
For premium (locked) videos, the backend generates a signed JWT token using the Cloudflare Stream signing key. The token is embedded in the HLS URL and expires after a short window (e.g. 1 hour). Without a valid signed token, Cloudflare will reject the playback request. This means even if someone discovers the playback URL, they cannot watch the video without a valid, unexpired token issued by the Tamkko backend.
3. Video Upload
3.1  Request Upload URL (Cloudflare Stream Direct Upload)
👤
User Story:  As a creator, I want to start uploading a video by requesting a secure direct upload URL from the server, so my video is sent straight to Cloudflare without passing through Tamkko's backend server.
POST
/api/v1/videos/upload/request
Request Upload URL
Auth Required
Yes — Authorization: Bearer {access_token}
Rate Limit
10 uploads per creator per hour
Note
This endpoint calls the Cloudflare Stream API server-side to generate a direct upload URL. No video data is sent to this endpoint — only metadata about the planned upload.
📤  Request Body
{
"file_size_bytes": 52428800,
"duration_seconds": 47,
"title": "Accra vibes this evening 🇬🇭",
"caption": "Just another beautiful evening in Accra. Tag someone who needs to see this!",
"hashtags": [
"accra",
"ghana",
"afrobeats",
"tamkko"
],
"visibility": "public",
"allow_comments": true,
"allow_duet": false
}
Field explanations:
file_size_bytes: size of the video file in bytes. Used by Cloudflare to pre-allocate the upload slot. Required.
duration_seconds: estimated video duration. Optional but recommended — helps with validation of max allowed duration.
visibility: public | followers_only | premium | private. Defaults to public if omitted.
allow_comments: whether viewers can post comments on this video. Default true.
allow_duet: reserved for Phase 3. Send false in Phase 1.
hashtags: array of strings, no # symbol, max 10 hashtags, max 30 chars each.
201
Created — Upload URL generated
✅  Response Body
{
"status": "success",
"message": "Direct upload URL generated. Upload your video file to the upload_url within 30 minutes.",
"data": {
"video_id": "vid_7k3m8n2p",
"cloudflare_uid": "ea95132c15732412d22c1476fa83f27a",
"upload_url": "https://upload.videodelivery.net/tus/ea95132c15732412d22c1476fa83f27a",
"upload_method": "TUS",
"upload_expires_at": "2025-03-01T10:30:00Z",
"max_duration_seconds": 180,
"status": "pending"
}
}
⚠️
The upload_url is a one-time TUS endpoint provided by Cloudflare Stream. The Expo app must upload the video binary directly to this URL using the TUS resumable upload protocol. The upload_url expires after 30 minutes. If it expires before the upload completes, the user must call this endpoint again to get a fresh URL.
400
Bad Request — Validation error
❌  Error Response
{
"status": "error",
"message": "Validation failed",
"errors": {
"file_size_bytes": [
"Video file size exceeds the 500 MB maximum"
],
"duration_seconds": [
"Video duration exceeds the 3-minute (180 second) maximum for Phase 1"
],
"hashtags": [
"Maximum 10 hashtags allowed"
],
"visibility": [
"Invalid visibility value. Must be: public, followers_only, premium, or private"
]
}
}
3.2  Confirm Upload & Save Video Metadata
👤
User Story:  As a creator whose video has finished uploading to Cloudflare, I want to confirm the upload so the backend knows the file was received and can begin tracking its processing status.
ℹ️
This endpoint is called by the Expo app immediately after the TUS upload to Cloudflare completes successfully. It tells the backend to begin polling Cloudflare for processing status. In most cases the Cloudflare Stream webhook (Section 2) will update the status automatically — this endpoint is the fallback confirmation trigger.
POST
/api/v1/videos/{video_id}/upload/confirm
Confirm Upload
Auth Required
Yes — Authorization: Bearer {access_token} (must be the creator who requested the upload)
📤  Request Body
{
"cloudflare_uid": "ea95132c15732412d22c1476fa83f27a"
}
200
OK — Upload confirmed, processing started
✅  Response Body
{
"status": "success",
"message": "Upload confirmed. Your video is being processed. This usually takes 1-3 minutes.",
"data": {
"video_id": "vid_7k3m8n2p",
"status": "processing",
"estimated_ready_in_seconds": 90
}
}

4. Video Feed
ℹ️
All feed endpoints use cursor-based pagination for infinite scroll. The next_cursor value from one response is passed as the cursor query parameter in the next request. When next_cursor is null, the feed has been exhausted. Never use page numbers for video feeds — cursors are more efficient and avoid duplicate videos when new content is inserted.

👤

This endpoint is called automatically by the Expo app in the background — not by the user. It should be called: (1) when the user swipes away from a video, and (2) every 10 seconds while a video is playing. Fire-and-forget — the app should not wait for a response before continuing. Rate Limit

"completed": false,
"source": "for_you_feed"
}

OK

6. User Search & Discovery
ℹ️
Search works in two stages — exactly like TikTok. Stage 1: As the user types, autocomplete suggestions appear instantly (debounced at 300ms). Stage 2: When the user presses Search or taps a suggestion, the full search results page loads with paginated results for both users and videos.
6.1  Search Autocomplete Suggestions
👤
User Story:  As a user typing in the search bar, I want to see instant autocomplete suggestions — usernames, display names, and hashtags — appearing as I type, so I can quickly tap the right result without finishing my query.
How it works: The frontend debounces this call at 300ms — it only fires if the user has stopped typing for 300 milliseconds. This prevents hammering the API on every keystroke. Results are cached in Redis for 30 seconds per query string so repeated common queries are served from cache.

Cache

All error responses follow the standard envelope. Use the code field for frontend error handling logic.

HTTP
Description

403

403
Video is private — only the creator can view it
VIDEO_PREMIUM_LOCKED
403
