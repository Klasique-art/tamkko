2. Registration
ℹ️
Registration on Tamkko is unified — all users (Ghana or international) register with the same form. There is no separate Ghana vs international flow. The user's country is captured during registration via a country selector, which informs the platform about their region for monetisation eligibility (e.g., Mobile Money is only available for Ghana accounts). Both phone and email are collected and must be verified.
2.1  Register (Phone + Email, Any Country)
👤
User Story:  As a new user anywhere in the world, I want to create a Tamkko account by providing my email, phone number, display name, username, country, and a password. I understand that anyone can be a content creator — there is no separate creator account type. Once registered, I can immediately start uploading videos or just watch content.
Field Explanations & Accepted Values:
country: ISO 3166-1 alpha-2 country code (e.g. GH for Ghana, NG for Nigeria, US for United States, GB for United Kingdom). This is selected from a country picker in the UI, not typed manually. It determines the user's region for features like Mobile Money payouts.
All users have the same account type. Any user can upload videos and receive tips — just like TikTok where any account can post content. There is no 'role' field. The platform automatically unlocks creator-specific features (wallet, tipping, withdrawals) once the user's phone is verified and they upload their first video.

3.3  Google OAuth 2.0 Login
👤
User Story:  As a user, I want to sign in with my Google account so I can skip manual password entry. If I am a new user, I will be asked to complete my profile (username and country) before my account is created.
How the Google OAuth flow works in Expo:
The app calls expo-auth-session with the Google OAuth provider. This opens the system browser (expo-web-browser) on the device.
The user logs in to Google and approves the Tamkko permissions request.
Google redirects back to the app via the deep link tamkko://auth/callback with an authorisation code.
expo-linking captures the deep link and passes the id_token and access_token to the frontend.
The frontend sends these tokens to the backend endpoint below.
The backend verifies the Google ID token, finds or creates the user, and returns a Tamkko JWT.

5. Two-Factor Authentication (2FA)
5.1  Complete 2FA Flow — Concept & Full User Journey
What is TOTP-based 2FA?
Tamkko uses TOTP (Time-Based One-Time Password) — the same standard used by Google, GitHub, and major banks. TOTP works by sharing a secret key between the server and an authenticator app. Every 30 seconds, both the server and the app independently generate the same 6-digit code from that secret. When the user types their code, the server checks it matches — no network call required by the authenticator app.
Do I need a second phone?
No. You do not need a second phone. The QR code is displayed on the Tamkko app on the user's phone, and the user scans it using an authenticator app (Google Authenticator or Authy) installed on the same phone or any other device. On mobile, the flow works as follows: the QR code is shown on screen, the user presses 'Open in Google Authenticator' (a deep link), which switches to the authenticator app pre-filled with the secret. Alternatively, the user can copy the text secret code manually and paste it into any authenticator app.
Complete Step-by-Step User Journey:
User navigates to Account Settings > Security > Enable Two-Factor Authentication.
The Tamkko frontend calls POST /api/v1/auth/2fa/enable (Section 5.2). The server generates a unique TOTP secret for this user and returns both the secret string and a QR code image URL.
The app displays the QR code image on screen with two options: (a) 'Open in Authenticator App' — a button that opens a deep link (otpauth://totp/...) which launches Google Authenticator or Authy pre-filled with the secret; (b) 'Copy setup key' — copies the plain-text secret so the user can manually add it in any authenticator app.
The user opens their authenticator app (on the same phone or another device) and adds the account. The authenticator app is now generating a new 6-digit code every 30 seconds.
The user returns to the Tamkko app and enters the current 6-digit code from their authenticator app into the confirmation screen.
The frontend sends this code to POST /api/v1/auth/2fa/verify-setup (Section 5.3). The server verifies the code, confirms the TOTP secret is correctly configured, enables 2FA on the account, and returns one-time backup codes.
The backup codes are displayed to the user ONE TIME ONLY. The user must save these somewhere safe (screenshot, password manager). They can be used if the user ever loses access to their authenticator app.
2FA is now active. From this point, every login requires the user to enter their password AND a 6-digit TOTP code from their authenticator app.
What happens if the user loses their phone / authenticator app?
The user can use one of their backup codes on the 2FA verification screen. Each backup code is single-use. After using a backup code the user is prompted to regenerate new ones. If all backup codes are also lost, the user must contact Tamkko support with identity verification to regain access.
Server-side validation window:
The server accepts codes from the current 30-second window and the one immediately before it (to account for clock skew between the user's device and the server). Codes from more than 60 seconds ago are rejected.
