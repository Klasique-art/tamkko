2. Tipping & Wallet — Architecture & Concepts
2.1  Money Flow Overview
ℹ️
All monetary values in the Tamkko API are in Ghana Cedis (GHS). Amounts are always represented as decimal numbers with exactly 2 decimal places (e.g. 10.00, 0.50). Never use integers for money. Never use JavaScript float arithmetic for money calculations — always use string-represented decimals from the API.
Complete money flow for a tip:
Viewer decides to tip a creator. Opens the tip bottom sheet and selects or types an amount (e.g. GHS 10.00). Enters their MTN MoMo number.
Expo app calls POST /api/v1/tips — backend creates a tip record (status: pending) and calls the Hubtel API to initiate a MoMo USSD push to the viewer's phone.
The viewer's phone receives a USSD prompt from MTN: 'Approve payment of GHS 10.00 to Tamkko? Press 1 to confirm.' The viewer approves on their phone.
MTN confirms the deduction to Hubtel. Hubtel sends a webhook callback to POST /api/v1/webhooks/hubtel with the payment result.
The backend webhook handler marks the tip as completed, credits the creator's wallet (minus the platform fee), and sends push notifications to both the viewer (tip sent confirmation) and the creator (you received a tip!).
The creator's wallet balance increases immediately. The funds are held as 'available_balance'. The creator can withdraw to their registered MTN MoMo account at any time above the minimum withdrawal threshold.
On withdrawal, the backend calls Hubtel's payout API to send GHS directly to the creator's MoMo number. The balance is debited and a withdrawal record is created.
2.2  MTN Mobile Money Integration (Hubtel Gateway)
Why Hubtel instead of the MTN MoMo API directly?
The MTN MoMo API requires direct partnership and certification with MTN Ghana — a lengthy process not suitable for Phase 1. Hubtel is a Ghanaian fintech company that provides a simplified REST API wrapping MTN MoMo, Vodafone Cash, and AirtelTigo Money. It handles the USSD push flow, reconciliation, and has sandbox/live environments. Using Hubtel means faster integration, no MTN certification needed, and support for multiple mobile wallets in Phase 2 with the same API.
Hubtel payment flow (technical detail):
Backend calls POST https://api.hubtel.com/v2/pos/onlinecheckout/direct-pay/mobile-money with the amount, customer MoMo number, reference (Tamkko tip ID), and callback URL.
Hubtel sends a USSD push to the customer's phone. The customer sees a prompt like: 'Confirm payment of GHS 10.00 to Tamkko. Enter PIN to approve.'
The customer enters their MoMo PIN on their phone. MTN processes the payment and notifies Hubtel.
Hubtel sends a POST request to the Tamkko backend callback URL (POST /api/v1/webhooks/hubtel) with the payment result: {Status: 'TXN_SUCCESS'} or {Status: 'TXN_FAILED'}.
The Tamkko backend verifies the Hubtel webhook signature, updates the tip/payment record status, and triggers the downstream actions (wallet credit, notifications).
⚠️
The USSD push prompt and PIN entry happen entirely on the customer's phone via MTN's network — Tamkko never sees or handles the customer's MoMo PIN. Tamkko only initiates the payment request and receives the result callback. This is the standard secure MoMo integration pattern.

Example: Viewer tips GHS 10.00. Platform takes GHS 1.50 (15%). Creator wallet is credited GHS 8.50. The Hubtel processing fee (~1.5% of the transaction) is absorbed by the platform from its 15% share.
2.4  Wallet Balance vs Pending vs Available
total_balance — The sum of all earnings ever credited to the wallet. Read-only historical figure.
pending_balance — Earnings from tips/subscriptions where the MoMo payment is still processing (status: pending or processing). These funds cannot be withdrawn yet.
available_balance — Funds that are fully settled and ready to withdraw. This is total_balance minus pending_balance minus already_withdrawn.
withdrawn_total — Total amount the creator has successfully withdrawn to their MoMo account over all time.
minimum_withdrawal — GHS 10.00. Creators cannot withdraw less than this amount.
maximum_withdrawal_per_day — GHS 2,000.00. Hubtel imposes this limit per account per day.
3. Tipping a Creator
ℹ️
Any authenticated user with a verified phone number can tip any creator on any video. Tipping is done in GHS. The minimum tip is GHS 1.00. There is no maximum tip limit in Phase 1. The tipper's MTN MoMo number is charged directly via USSD push.

3.2  MoMo Payment Callback (Webhook — Hubtel to Backend)
👤
User Story:  As the system, when Hubtel notifies the backend that a MoMo payment was approved or rejected by the user, I need to update the tip status, credit the creator's wallet, and send push notifications to both parties.
ℹ️
This endpoint is called by the Hubtel payment gateway — NOT by the Expo frontend. The frontend should never call this directly. The backend must verify the Hubtel-Signature header on every incoming webhook before processing.
POST
/api/v1/webhooks/hubtel
Hubtel Payment Callback
Auth Required
No — but must verify Hubtel-Signature header using HMAC-SHA256
Security
Backend verifies signature: HMAC-SHA256(payload_body, HUBTEL_WEBHOOK_SECRET). Reject any request where signature does not match.
Idempotency
Hubtel may send the same webhook more than once. The backend must be idempotent — if the tip is already 'completed', ignore duplicate callbacks.
📥  Hubtel Webhook Payload (Success)
{
"ResponseCode": "0000",
"Status": "TXN_SUCCESS",
"Data": {
"TransactionId": "HBT-20250301-001234",
"ClientReference": "tip_4m5n6p7q",
"Amount": 10,
"Charges": 0.15,
"AmountAfterCharges": 9.85,
"PhoneNumber": "0241234567",
"Description": "Tamkko tip payment",
"CreatedAt": "2025-03-01T10:02:30Z"
}
}
📥  Hubtel Webhook Payload (Failed / Rejected)
{
"ResponseCode": "1001",
"Status": "TXN_FAILED",
"Data": {
"TransactionId": "HBT-20250301-001235",
"ClientReference": "tip_4m5n6p7q",
"Amount": 10,
"Description": "User cancelled payment",
"CreatedAt": "2025-03-01T10:03:00Z"
}
}
Backend processing on TXN_SUCCESS:
Look up the tip record by ClientReference (tip_id).
Verify Amount matches the tip amount in the database — reject if mismatched.
Mark tip status as 'completed'.
Credit creator's wallet: available_balance += (tip_amount_ghs × 0.85). Record the transaction.
Record platform revenue: platform_revenue += (tip_amount_ghs × 0.15).
Send push notification to creator: 'You received a GHS 10.00 tip from @viewer on your video!'
Send push notification to tipper: 'Your GHS 10.00 tip to @creator was sent successfully!'
Increment the video's tips_count stat.
Backend processing on TXN_FAILED:
Mark tip status as 'failed' with failure_reason from the webhook.
Do NOT credit the creator's wallet.
Send push notification to tipper: 'Your tip payment was not completed. No charge was made.'

4. Creator Subscriptions (Premium Content Access)
ℹ️
When a creator marks videos as 'premium', viewers must subscribe to that creator to access all their premium content. A subscription is a recurring monthly charge — not a per-video payment. Once subscribed, the viewer can watch all of the creator's premium videos for the full billing month. The viewer can cancel at any time; access continues until the end of the paid period. The payment flow uses MoMo USSD push via Hubtel, identical to tipping. Each creator sets their own monthly subscription price.

6. Withdrawals
ℹ️
Withdrawals send the creator's available balance directly to their registered MTN MoMo account via Hubtel's payout API. Withdrawals are processed automatically in real time during business hours (6 AM – 10 PM GMT). Requests outside these hours are queued and processed at 6 AM the next day. There is no withdrawal fee — creators receive the full available amount they request.
6.1  Add / Update Mobile Money Account
👤
User Story:  As a creator, I want to register my MTN MoMo number as my withdrawal account so the platform knows where to send my earnings. I can only have one MoMo account registered at a time. Changing it requires OTP verification.
PUT
/api/v1/wallet/momo-account
Save MoMo Account
Auth Required
Yes — Authorization: Bearer {access_token}
OTP Required
Yes — an OTP must be sent to the new MoMo number first (purpose: phone_change) and verified. Include the action_token from the OTP verify response.
Note
Only one MoMo account per wallet. Saving a new one replaces the old one. Previous withdrawals are not affected.

8. Error Code Reference
All error responses follow the standard envelope. The code field is used for frontend error handling logic and should never be shown raw to users.
Error Code
HTTP
Description
TIP_NOT_FOUND
404
Tip with the given ID does not exist
CANNOT_TIP_OWN_VIDEO
403
A creator cannot tip their own video
CANNOT_SUBSCRIBE_TO_SELF
403
A creator cannot subscribe to their own channel
PHONE_NOT_VERIFIED
403
Phone number not verified — wallet features locked
NO_MOMO_ACCOUNT
400
No MoMo account registered — add one before withdrawing
INSUFFICIENT_BALANCE
400
Available balance is less than the requested withdrawal amount
BELOW_MINIMUM_WITHDRAWAL
400
Withdrawal amount is below the GHS 10.00 minimum
ABOVE_DAILY_LIMIT
400
Withdrawal exceeds the GHS 2,000.00 daily Hubtel limit
MOMO_PAYMENT_FAILED
400
MTN MoMo payment was rejected, cancelled, or timed out
MOMO_NUMBER_INVALID
400
Phone number is not a valid MTN Ghana MoMo number
MOMO_PROVIDER_UNSUPPORTED
400
Only MTN MoMo is supported in Phase 1
ALREADY_SUBSCRIBED
409
Viewer already has an active subscription to this creator — returns existing subscription details
CREATOR_NOT_SUBSCRIBED
403
Viewer does not have an active subscription to access this creator's premium content
SUBSCRIPTION_NOT_FOUND
404
Subscription record with the given ID does not exist
PLAYBACK_TOKEN_EXPIRED
401
Cloudflare signed playback token has expired — request a fresh one
WITHDRAWAL_NOT_FOUND
404
Withdrawal record not found

WEBHOOK_SIGNATURE_INVALID
403
Hubtel webhook signature verification failed — request rejected
WITHDRAWAL_FLAGGED
403
Withdrawal flagged for manual review — pending admin approval
WALLET_NOT_FOUND
404
Wallet record not found for this user
INTERNAL_SERVER_ERROR
