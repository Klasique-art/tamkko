1. Technology Stack & Dependencies
Tamkko uses Expo Push Notifications as the single push delivery service for both iOS and Android. Expo acts as a unified gateway — the backend sends one API call to Expo's push service, which then fans out to APNs (Apple) or FCM (Google Firebase) as appropriate for each device. This avoids managing two separate push credentials in Phase 1.

2.2  Expo Push Notification Flow
Step-by-step from trigger event to device screen:
An event occurs in the app (e.g. someone tips a creator). The Django view or Celery task that handles the event calls the notification service.
The notification service: (a) creates an in-app notification record in the DB for the recipient, (b) looks up all active Expo push tokens for the recipient from the device_tokens table, (c) dispatches a Celery task to send the push notification asynchronously.
The Celery task calls the Expo Push API: POST https://exp.host/--/api/v2/push/send with the push token, title, body, data payload (includes type and navigation target), and badge count.
Expo routes the message to APNs (iOS) or FCM (Android) based on the token prefix. The OS delivers the notification to the device.
The user taps the notification. The Expo app receives the tap event in the notification response handler. It reads the data.screen and data.params fields and calls router.push() to navigate to the correct screen.
The app calls PATCH /api/v1/notifications/{notification_id}/read to mark the in-app notification as read and decrement the badge count.
⚠️
Never send push notifications synchronously inside a Django request/response cycle. Always dispatch to Celery. A slow or failed push send must never cause the triggering API call (e.g. tip confirmation) to fail or time out.

3. Device Token Management
ℹ️
An Expo Push Token is a unique string (format: ExponentPushToken[xxxxxx]) that identifies a specific app installation on a specific device. It must be registered with the backend after login so the backend knows which device to send pushes to. A single user can have multiple tokens (phone + tablet, or after reinstall). The backend stores all active tokens per user and sends to all of them.
3.1  Register Push Token
👤
User Story:  As a user who just logged in or opened the app, I want my device's Expo push token to be registered with the backend so I start receiving push notifications for activity on my account.
When to call: Call this endpoint on every app launch after login, not just on first install. Tokens can change after app updates or OS reinstalls, so always register the current token on launch.
POST
/api/v1/notifications/token
Register Push Token
Auth Required
Yes — Authorization: Bearer {access_token}
Idempotent
Yes — registering the same token twice is safe. The backend upserts by token string.
📤  Request Body
{
"expo_push_token": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]",
"device_type": "ios",
"device_model": "iPhone 15 Pro",
"app_version": "1.0.0"
}
device_type accepted values: ios | android
200
OK — Token registered
✅  Response Body
{
"status": "success",
"message": "Push token registered.",
"data": {
"token_id": "dtkn_1a2b3c4d",
"expo_push_token": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]",
"device_type": "ios",

}
}
3.2  Deregister Push Token (Logout / Permission Revoked)
👤
User Story:  As a user logging out or revoking push permission, I want my device token removed from the backend so I stop receiving notifications on this device.
When to call: (1) On explicit logout. (2) When expo-notifications returns null for the push token — this means the user revoked permission. Always deregister before clearing the local auth token.
DELETE
/api/v1/notifications/token
Deregister Token
Auth Required
Yes — Authorization: Bearer {access_token}

{
"expo_push_token": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]"
}

OK
✅  Response Body
{
"status": "success",
"message": "Push token deregistered. You will no longer receive notifications on this device."
}

6. Admin — Notifications
ℹ️
Admin notification endpoints allow the platform team to send broadcast messages to all users or targeted messages to specific users — for example announcing a new feature, warning a creator about policy violations, or notifying a user of an account action.
6.1  Send a Broadcast Notification (All Users)
👤
User Story:  As an admin, I want to send a push notification and in-app notification to all Tamkko users at once — for example to announce a new feature or a major event.
POST
/api/v1/admin/notifications/broadcast
Broadcast Notification
Auth Required
Yes — Admin JWT
Delivery
Queued as a Celery task. Large user bases are sent in batches of 100 tokens (Expo API limit). The response returns immediately — actual delivery happens in the background.
Rate Limit
1 broadcast per hour. Prevents accidental spam to all users.
